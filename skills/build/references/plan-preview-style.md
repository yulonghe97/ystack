# Plan Preview Style Reference

Use this reference when generating `.context/<feature-id>/plan.html` during `/build`.

The preview is a generated review surface for humans. It is not canonical. `DECISIONS.md` and `PLAN.md` remain the source of truth.

## Template

Use `skills/build/templates/plan-preview.html` as the starting point. **Copy the template file** to `.context/<feature-id>/plan.html` and then use `Edit` to replace each placeholder in place. Do not regenerate the file end-to-end with `Write` — that wastes tokens duplicating the static skeleton on every plan. See SKILL.md Phase 6 for the exact `cp` + `Edit` workflow.

Required placeholders:

- `{{FEATURE_TITLE}}` - escaped plain text title (also appears inside `<title>` prefixed with `ystack · /build preview · `)
- `{{FEATURE_SUMMARY}}` - escaped plain text summary
- `{{APPROVAL_STATE}}` - usually `Awaiting Approval`
- `{{SUMMARY_CHIPS}}` - chip strip rendered directly under the header. Must include count chips for success criteria and tasks, plus one chip per applicable impact category. Omit impact chips when no impact category applies.
- `{{CONTEXT_SECTION}}` - rendered HTML section
- `{{DECISIONS_AND_CRITERIA_SECTION}}` - rendered HTML section
- `{{IMPACT_SECTIONS}}` - zero or more rendered impact sections
- `{{TASKS_SECTION}}` - rendered HTML section
- `{{FOOTER_NOTE}}` - rendered note prefixed with `Generated by the ystack /build skill.` followed by the per-feature note
- `{{DECISIONS_MARKDOWN_HTML}}` - rendered Markdown from `DECISIONS.md`
- `{{PLAN_MARKDOWN_HTML}}` - rendered Markdown from `PLAN.md`

## Visual Rules

- Use the template's flat blueprint style: bordered sections, white canvas, slate borders, compact typography.
- Do not add shadows, gradients, rounded app chrome, marketing layout, decorative backgrounds, or editing controls.
- Do not add a top stepper, top flow strip, table of contents, or duplicate row summarizing the section order.
- The `Visual Review` tab starts directly with `01 / Context`.
- Use monospace only for paths, commands, task ids, badges, ASCII graphs, and technical values.
- Keep sections dense enough for review. This is an approval page, not a landing page.

## Sticky Tabs

The `.tabs` element uses `position: sticky; top: 0; z-index: 2;` with an opaque background, so the tab bar stays in reach while the reviewer scrolls a long visual review. On viewports below 900px the tabs revert to `position: static` to preserve narrow-viewport readability. Do not introduce JavaScript to mimic this — sticky positioning is plain CSS.

## Summary Strip

A `.summary-strip` row sits directly between the `<header>` and the `<div class="content">`, populated from the `{{SUMMARY_CHIPS}}` placeholder. It is always visible regardless of which tab is active.

Allowed content:

- Count chips for success criteria and tasks (e.g., `8 CRITERIA`, `3 TASKS`).
- One chip per applicable impact category present in the plan (e.g., `MIGRATIONS`, `CRITICAL CHANGES`, `ENV VARS`, `DOCS TO UPDATE`).

Rules:

- Omit impact chips when no impact category applies; do not render `NONE` chips.
- Use existing tokens (`--c-border`, `--c-soft`, mono labels). No new colors, no icons, no shadows.
- The strip is not sticky; only the tab bar is.

## Identity Strings

Every generated preview must carry these fixed identity strings — not the per-feature title:

- `<title>` is `ystack · /build preview · {{FEATURE_TITLE}}`.
- The header eyebrow text is `ystack · /build preview` (the template ships with this; do not replace it with `Visual approval layer` or anything else).
- The `{{FOOTER_NOTE}}` content is prefixed with `Generated by the ystack /build skill.` followed by the per-feature note.

This is identity, not branding chrome — no logo image, no color change, same flat blueprint style.

## Tabs

The preview has exactly these review tabs:

- `Visual Review`
- `DECISIONS.md`
- `PLAN.md`

The Markdown tabs must show rendered Markdown, not raw source text. Convert Markdown to HTML at generation time and inline it into the template. Do not use browser-side `fetch()` for sibling Markdown files because local `file://` behavior is unreliable.

Escape unsafe HTML from Markdown content before rendering or only support a safe Markdown subset.

## Visual Review Sections

Use this section order:

- `01 / Context`
- `02-03 / Decisions + Success Criteria`
- `04 / Impact Review`, only when at least one impact category applies
- `05 / Tasks`

Do not show empty sections. If no impact category applies, omit `Impact Review`.

## Success Criteria

Render success criteria as a table or fixed-column checklist.

Do not render `[ ]` as raw text in the visual review. The checkbox marker must stay in a fixed narrow column and the criterion text must wrap independently.

## Impact Review

Render each applicable impact item as its own section inside `Impact Review`.

Supported impact categories:

- `Architectural Graph` - module boundaries, new dependencies, or data flow changes
- `Critical Changes` - behavioral, security, data, billing, auth, compatibility, or user-visible risks
- `Environment Variables Added` - new keys, required environments, defaults, and secret requirements
- `Migrations` - schema migrations, backfills, data risks, and rollback notes
- `Docs To Be Updated` - exact docs pages that `/docs` must update after implementation is verified

If a category does not apply, omit it. Do not render `None` cards in real generated previews.

## ASCII Graphs

ASCII graphs are allowed when they clarify sequencing, architecture, or execution waves. They should be secondary to structured sections and must be inside relevant sections, not in a separate decorative block.

Use ASCII only:

```text
task-1
  |
  +-- task-2
  |
  +-- task-3
```

## Source Of Truth

The preview must not introduce a decision, criterion, task, migration, environment variable, or doc update that is missing from `DECISIONS.md` or `PLAN.md`.

If the preview reveals missing content, update the Markdown first, then regenerate the preview.
