# Plan Preview Checklist

Run this check before opening `.context/<feature-id>/plan.html`.

## Required Files

- [ ] `.context/<feature-id>/DECISIONS.md` exists.
- [ ] `.context/<feature-id>/PLAN.md` exists.
- [ ] `.context/<feature-id>/plan.html` exists.
- [ ] `plan.html` was produced by copying `skills/build/templates/plan-preview.html` and `Edit`-ing placeholders in place — not by writing the file end-to-end with `Write`.

## Template Structure

- [ ] The preview has exactly these tabs: `Visual Review`, `DECISIONS.md`, `PLAN.md`.
- [ ] `Visual Review` starts directly with `01 / Context`.
- [ ] There is no top flow strip, stepper, table of contents, or duplicate `Context -> Decisions -> Criteria -> Tasks` row.
- [ ] `DECISIONS.md` and `PLAN.md` tabs show rendered Markdown, not raw source in a `<pre>` block.
- [ ] The preview does not use browser-side `fetch()` to load Markdown files.

## Visual Review

- [ ] Context identifies feature intent, module, canonical inputs, and generated preview path.
- [ ] Decisions and success criteria are visible before tasks.
- [ ] Success criteria use a fixed checkbox column or table layout.
- [ ] Checkbox markers do not split or wrap separately from the criterion text.
- [ ] Task order, dependencies, file targets, and verification steps are visible.
- [ ] Do/Verify field text in task cards renders at body scale (≈12–13px), not the 16px browser default.

## Sticky Tabs and Summary Strip

- [ ] The tab bar stays visible while scrolling a long visual review (sticky to viewport top on desktop viewports).
- [ ] On viewports below 900px the tab bar reverts to non-sticky and remains scrollable.
- [ ] The summary strip sits directly between the header and the content, always visible regardless of active tab.
- [ ] The summary strip shows the expected counts (criteria, tasks) plus one chip per applicable impact category — and only categories that actually apply (no `NONE` chips).

## Identity

- [ ] The document `<title>` is `ystack · /build preview · {{FEATURE_TITLE}}`.
- [ ] The header eyebrow reads `ystack · /build preview`.
- [ ] The footer note is prefixed `Generated by the ystack /build skill.` followed by the per-feature note.

## Impact Review

- [ ] `Architectural Graph` appears only if architecture, dependency, module boundary, or data flow changes exist.
- [ ] `Critical Changes` appears only if large behavioral, security, data, billing, auth, compatibility, or user-visible changes exist.
- [ ] `Environment Variables Added` appears only if new environment variables are required.
- [ ] `Migrations` appears only if schema migrations, backfills, data risks, or rollback notes exist.
- [ ] `Docs To Be Updated` appears only if exact docs pages need updates after verified implementation.
- [ ] No empty impact category renders as `None`.

## Source Consistency

- [ ] Every visible decision appears in `DECISIONS.md`.
- [ ] Every visible success criterion appears in `PLAN.md`.
- [ ] Every visible task appears in `PLAN.md`.
- [ ] Every visible impact item is represented in `DECISIONS.md` or `PLAN.md`.
- [ ] If the preview revealed missing content, the Markdown was updated first and the preview regenerated.

## Style

- [ ] The page uses flat bordered sections from the template.
- [ ] There are no shadows, gradients, marketing hero sections, editing controls, or app navigation.
- [ ] ASCII graphs, if present, are inside relevant sections and use ASCII characters only.
- [ ] The page remains readable on a narrow mobile viewport.
